# Phase 4: 虫族 Zerg

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 添加虫族 Zerg 单位、建筑、幼虫系统，扩展 UnitType 和 BuildingType，更新渲染颜色区分三个种族。

**Architecture:** 幼虫机制由 Building 组件扩展属性驱动（`larvae: number`），训练系统检查 `larvae > 0` 后才允许生产，建筑提供不同颜色几何体。种族颜色：人族蓝 (0x00aaff)、虫族紫 (0x8800ff)、神族金黄 (0xffcc00)。

**Tech Stack:** Three.js + React + ECS + TypeScript strict + Vitest

## Task 2: 种族颜色渲染

**Files:**
- Modify: `src/renderer/EntityRenderer.ts` — 按 ownerId 区分种族颜色

**当前逻辑:**
```typescript
const color = unit.ownerId === 0 ? 0x00aaff : 0xff4422;
```

**修改为三族颜色:**
```typescript
private raceColor(ownerId: number): number {
  if (ownerId === 0) return 0x00aaff;      // 人族蓝
  if (ownerId === 1) return 0x8800ff;     // 虫族紫
  if (ownerId === 2) return 0xffcc00;     // 神族金黄
  return 0xaaaaaa;                          // 默认灰
}
```

建筑也用同样逻辑：人族蓝(0x00aaff)、虫族紫(0x8800ff)、神族金黄(0xffcc00)。建筑颜色从 Building 组件的 buildingType 判断：hatchery=虫族、Lair/Hive=虫族、gateway/pylon=神族，其余人族。

或者更简单的方案：从 Unit.ownerId 推算（ownerId=0 玩家人族，ownerId=1 AI 虫族，ownerId=2 神族）。但实际上 ownerId 只是 playerId，不一定等于种族。更好的方案：添加 `Player` 组件记录 `race: 'terran' | 'zerg' | 'protoss'`。但这超出 Task 2 范围。

**简化方案（Task 2 范围内）:**
添加 `ownerRace` 推断函数：ownerId % 3 → 0:terran, 1:zerg, 2:protoss。或者直接按 ownerId=1 时渲染紫色（AI 默认为虫族）。

```typescript
const RACE_COLORS: Record<string, number> = {
  terran: 0x00aaff,
  zerg: 0x8800ff,
  protoss: 0xffcc00,
};

private unitColor(ownerId: number): number {
  // 简单：ownerId 模3映射种族
  // ownerId=0 → 人族（玩家）, ownerId=1 → 虫族（AI）
  if (ownerId === 0) return RACE_COLORS.terran;
  if (ownerId === 1) return RACE_COLORS.zerg;
  if (ownerId === 2) return RACE_COLORS.protoss;
  return 0xaaaaaa;
}
```

建筑也用同样颜色（根据建造者的 ownerId）。

## Task 6: Zerg 建筑建造

**Files:**
- Modify: `src/systems/BuildSystem.ts` — 扩展支持 Zerg 建筑
- Modify: `src/data/buildings.json` — 确认 Zerg 建筑 cost/supply 数据正确

**当前 BuildSystem.placeBuilding 检查 `playerResources.canAfford(cost.minerals, cost.supply)`。**

Zerg 的 Overlord 提供负 supply（增加人口上限）：`cost.supply: -8`。

**问题：** `PlayerResources.useSupply(amount)` 接受负数吗？当前实现是 `supplyUsed += amount`。如果 amount=-8，则 supplyUsed -= 8（减少使用量，等于增加上限）。

需要验证：
```typescript
// 当前 PlayerResources.useSupply:
public useSupply(amount: number): void {
  this.supplyUsed += amount;
}
```
如果 `supplyUsed = 0`，调用 `useSupply(-8)` → `supplyUsed = -8`。这在 UI 显示 "0 / 10" 时不对。正确的处理：负 supply 应该增加 supplyMax，而不是减少 supplyUsed。

**修复：** 在 `BuildSystem.placeBuilding` 中，如果 `supply < 0`，改为增加 `supplyMax`：
```typescript
if (data.supply < 0) {
  this.playerResources!.supplyMax += Math.abs(data.supply);
} else {
  this.playerResources!.useSupply(data.supply);
}
```
